# TravelBlock PRD: v0.10.0-alpha Seat & Ticket Rewards System

## Summary
TravelBlock already simulates flights, captures seat selection, and stores enough flight metadata to support progression. What it does not have yet is a durable rewards layer that turns completed flights into points, lets users spend those points on better seats or ticket tiers, and gives them a simple place to inspect their balance and transaction history.

This release adds a local-only points economy and a lightweight rewards storefront that sits naturally on top of the existing flight loop.

## Problem Statement
The current app makes every flight feel isolated:

1. A completed flight does not yet translate into a persistent reward the user can spend later.
2. Seat selection already exists, but `pointsCost` is always `0` in `utils/flightGenerator.ts`, so there is no meaningful economy behind it.
3. There is no balance UI, no transaction history, and no checkout flow for upgrading seats or ticket tiers.
4. Without a rewards layer, the app has less retention pressure between sessions and less reason for users to keep coming back.

The feature should feel like progression, not gambling or monetization noise.

## Current State
The codebase already provides a strong foundation for this feature:

- `types/flight.ts` already defines `Seat`, `SeatClass`, `FlightBooking`, `BoardingPass`, and `CompletedFlight`.
- `utils/flightGenerator.ts` already creates cabin layouts and includes a `pointsCost` field on each generated seat.
- `components/flight/SeatMap.tsx` already renders seat class distinctions and can support paid seat metadata without a full redesign.
- `app/flight/seat-selection.tsx` already owns the seat-picking step where an upgrade or paid preference could be offered.
- `app/flight/booking-confirmation.tsx`, `app/flight/check-in.tsx`, `app/flight/boarding-pass.tsx`, and `app/flight/cockpit.tsx` already form the end-to-end journey that can trigger reward earning.
- `context/FlightContext.tsx` persists active flight state, so the rewards layer can piggyback on the existing app state architecture.
- `services/storageService.ts` and `expo-sqlite/kv-store` already provide local persistence primitives.
- `app/(tabs)/index.tsx` is the clearest home-screen entry point for a rewards balance card or store shortcut.
- `app/settings.tsx` already exists as a secondary utility route for account-style controls.
- `ROADMAP.md` explicitly reserves `v0.10.0-alpha` for the rewards economy.

## Goals
- Add a durable local points balance and transaction ledger.
- Award points automatically when a flight completes.
- Let users spend points on ticket tiers and seat preferences.
- Make the current seat-selection flow support paid options without breaking free random seat assignment.
- Provide a simple balance summary and transaction history view.
- Keep the experience mobile-first, fast, and easy to understand.
- Preserve the existing flight flow and avoid turning the app into a commerce-first product.

## Non-Goals
- Do not add real-money payments.
- Do not add cloud sync or multi-device account migration.
- Do not add social leaderboards or public profiles.
- Do not require an internet connection for any core rewards action.
- Do not redesign the cockpit or booking flow from scratch.
- Do not introduce ads, loot boxes, or casino-like mechanics.

## User Stories
1. As a user, I want to earn points when I complete a flight so the app feels cumulative.
2. As a user, I want to see my current balance so I know what I can afford.
3. As a user, I want to spend points on better seats or ticket tiers so I can customize my trip.
4. As a user, I want to review my point history so I understand how I earned and spent points.
5. As a user, I want free random seat selection to remain available so the app still works without spending points.

## Functional Requirements

### 1. Points Ledger
- The app must maintain a local points balance.
- The app must record every award and spend as a transaction.
- Every transaction must include:
  - unique transaction id
  - type (`earn` or `spend`)
  - amount
  - reason
  - timestamp
  - optional reference id to the completed flight or purchase
- The ledger must be append-only by default.
- Spending must be atomic and must fail if the balance is insufficient.
- The balance must persist across app restarts.

### 2. Points Earning
- The app must award points when a flight completes.
- The award should be derived from flight metadata already available in `CompletedFlight`:
  - distance
  - duration
  - diverted vs non-diverted status
  - seat class or cabin tier if relevant
- The award logic must be deterministic and testable.
- The awarding path must be idempotent so the same completed flight cannot credit points twice.
- The user should receive brief feedback that points were earned, but the reward should not block the post-flight experience.

### 3. Ticket Store
- The app must provide a rewards store where users can inspect ticket tiers and seat upgrade options.
- The store must show:
  - current balance
  - item price in points
  - item description or benefit
  - whether the item is currently purchasable
- The store must support at least:
  - cabin tier upgrades where the aircraft supports them
  - preferred seat options such as window, aisle, or front-row preferences
- Random seat assignment must remain free and available.
- If a user cannot afford an item, the UI must clearly show why the purchase is disabled.
- Checkout must show a confirmation state before committing a spend transaction.

### 4. Seat Selection Integration
- The current seat selection screen must continue to support free seat choice.
- Paid seat options must be represented without breaking the existing `SeatMap` UI.
- If a seat carries a points cost, the user must see that cost before confirming.
- If the user purchases a premium seat, the selected seat must persist into boarding pass generation and cockpit display.
- If the user chooses a free seat, the flow must behave exactly as it does today.

### 5. Balance and History UI
- The app must expose a clear points summary view.
- The summary must show:
  - current balance
  - points earned from recent flights
  - recent spends
- The app must expose a transaction history view.
- The history must be readable on small screens and support scroll performance for a growing ledger.
- Empty states must explain how points are earned.

### 6. Navigation and Routing
- The rewards experience must be reachable through Expo Router routes, not hidden behind ephemeral state.
- The Home tab should expose the primary entry point to rewards.
- The Settings route can expose secondary account-style controls such as history, receipts, or pricing rules.
- The root stack must register any new rewards/store/history routes.
- Navigation from flight completion into rewards must not create back-stack loops that return the user to an ended cockpit.

## UX Requirements
- Keep the visual language aviation-themed and calm, not neon or arcade-like.
- Make the store feel like a travel upgrade screen, not a casino.
- Keep copy short and concrete.
- Use clear price labels and purchase confirmations.
- Support both light and dark themes.
- Make the balance visible without overpowering the main flight experience.
- Preserve large touch targets and one-handed usability on mobile.

## Data and State Requirements
- Keep points state separate from active flight state.
- Persist the ledger locally using SQLite-backed storage.
- Define a stable record shape for rewards transactions so future analytics can consume it.
- Use idempotency keys or completed-flight references so repeated completion events do not double-award points.
- Never allow a spend transaction to drive the balance below zero.
- Preserve historical transactions even if the user changes home airport, seat preference, or flight setup state.

## Success Metrics
- Users receive points after completed flights without manual action.
- Users can inspect their balance in two taps or fewer from the home screen.
- Users can complete a purchase flow for at least one seat or ticket upgrade.
- Duplicate awards are effectively eliminated.
- The app remains smooth with a growing history of rewards transactions.
- The feature increases the number of users who start a second flight in the same session.

## Dependencies
- `app/flight/cockpit.tsx` or the post-flight handoff must expose a completed-flight event for awarding points.
- `types/flight.ts` remains the source of truth for completed-flight data.
- `utils/flightGenerator.ts` and `components/flight/SeatMap.tsx` must continue exposing seat metadata.
- `context/FlightContext.tsx` must continue persisting seat and boarding pass state so purchases flow through the current booking model.
- `services/storageService.ts` and `expo-sqlite/kv-store` provide local persistence foundations.
- `app/(tabs)/index.tsx` and `app/settings.tsx` are the most likely UI entry points for rewards navigation.

## Risks
- If the reward formula feels too generous, progression will flatten quickly.
- If the reward formula feels too stingy, the feature will feel grindy and pointless.
- If point earning is not idempotent, repeated completion events could inflate balances.
- If premium seat pricing is too visually prominent, the app may feel pay-to-win.
- If store navigation is buried too deeply, users may never discover the feature.
- If purchase state is stored only in memory, users may lose selections on app backgrounding.

### Phase 1: Points Foundation
- [ ] Add a local points ledger and balance service.
- [ ] Define transaction and balance types.
- [ ] Implement deterministic award and spend operations.
- [ ] Add tests for earning rules, insufficient funds, and idempotency.

### Phase 2: Rewards Store
- [ ] Add a routed ticket store screen.
- [ ] Show balance, prices, and purchasable items.
- [ ] Add confirmation and failure states for checkout.
- [ ] Add tests for rendering and purchase behavior.

### Phase 3: Seat Upgrade Integration
- [ ] Connect paid seat metadata into the existing seat selection flow.
- [ ] Preserve free random-seat behavior.
- [ ] Ensure boarding pass and cockpit state reflect the purchased seat.
- [ ] Add tests for paid vs free seat paths.

### Phase 4: Completion Awarding and History
- [ ] Award points from flight completion using a completed-flight reference.
- [ ] Surface a points summary and transaction history view.
- [ ] Add navigation entry points from the Home and Settings screens.
- [ ] Add tests for post-flight awarding and history rendering.

## Testing Requirements
- Unit test the point formula and transaction validation rules.
- Unit test balance updates, spend failures, and duplicate-award protection.
- Test that a completed flight creates exactly one earn transaction.
- Test that the store renders correctly when the balance is zero and when it is positive.
- Test the free-seat path and the paid-seat path separately.
- Test navigation from Home to the rewards store and from the store to history.
- Verify theme support in light and dark mode.
- Manually validate the checkout flow on a physical device because this is a mobile-first UX.

## Release Notes Copy
TravelBlock now rewards completed flights. Earn points when you land, spend them on better seats or ticket tiers, and review your balance and history in a new rewards experience.

## Open Questions
1. Should the first release price only seat preferences, or should it also allow cabin-tier upgrades where the aircraft supports them?
2. Should points be awarded primarily from distance, duration, or a mix of both?
3. Should the rewards entry point live on the Home tab, inside Settings, or in a dedicated route subtree?
4. Should premium seat purchases be one-time per flight or persistent preferences for future bookings?

## Out of Scope for This Release
- Real-money checkout
- Subscriptions
- Cloud-synced wallets
- Social leaderboards
- Shared family accounts
- Ads or promotional offers
