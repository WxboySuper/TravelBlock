# TravelBlock PRD: v0.8.0-alpha Post-Flight Pivot Flow

## Summary
TravelBlock currently ends every completed flight with a cockpit alert. That is a functional stopping point, but it is not a product experience. This feature replaces the abrupt arrival alert with a dedicated post-flight flow that lets the user review the flight, celebrate landing, and choose what happens next.

The new flow must support two clear next steps:

1. Continue Journey: treat the airport just landed at as the next temporary origin.
2. Return Home: route the user back to the saved home airport and prepare the next flight from there.

This release is the bridge between the live cockpit simulation and the future logbook/history system in v0.9.0.

## Problem Statement
The current cockpit completion path in `app/flight/cockpit.tsx` resolves to a blocking alert with a single return action. That creates three product problems:

1. Users do not get a proper landing moment or summary of what they completed.
2. Users cannot chain flights naturally from the airport they just reached.
3. The app does not yet capture completed-flight data in a reusable shape for the logbook work that follows.

The experience should feel like an arrival, not just a timer reaching zero.

## Current State
The app already has most of the flight journey in place:

- `app/flight/setup.tsx` and `app/flight/destination.tsx` handle pre-flight selection.
- `app/flight/review.tsx`, `app/flight/seat-selection.tsx`, `app/flight/booking-confirmation.tsx`, `app/flight/check-in.tsx`, and `app/flight/boarding-pass.tsx` complete the booking and boarding flow.
- `app/flight/cockpit.tsx` runs the real-time timer, map, metrics, and divert logic.
- `services/flightTimerService.ts` emits arrival events.
- `types/flight.ts` already defines `CompletedFlight`.
- `context/FlightContext.tsx` persists origin, destination, duration, booking, seat, and boarding pass, but it does not yet distinguish home airport from the current post-flight origin.
- Expo Router is already used for file-based navigation, so the new arrival flow should be added as a real route rather than an ad hoc modal or alert.

Official Expo Router guidance reinforces this route-first model: files in `app/` become routes automatically, and the stack can be organized through `Stack.Screen` entries in the root layout.

## Goals
- Replace the arrival alert with a dedicated landed screen.
- Show the airport just reached and a concise route summary.
- Surface key completion stats in a mobile-friendly layout.
- Let the user continue from the landed airport without changing their saved home airport.
- Let the user return home in one tap.
- Capture a serializable completed-flight payload that v0.9.0 can consume without redesign.
- Keep the flow stable under Expo Router navigation and mobile app backgrounding.

## Non-Goals
- Do not build the logbook UI, analytics, or charts.
- Do not implement rewards, points, or seat monetization.
- Do not redesign the pre-flight booking flow.
- Do not change the core timer, divert, or map simulation mechanics.
- Do not add a new transport mode.

## User Stories
1. As a user, when a flight ends, I want to see where I landed and what I accomplished.
2. As a user, I want to continue my journey from the airport I just reached.
3. As a user, I want to return to my saved home airport with one tap.
4. As a user, I want the app to remember the flight I just completed so future history features can use it.

## Functional Requirements

### 1. Landed Screen
- The app must navigate to a dedicated landed screen when the cockpit timer completes.
- The landed screen must show:
  - arrival airport name
  - airport code
  - city and country
  - origin to destination route summary
- The landed screen must show completion stats at minimum:
  - distance flown
  - total flight time
  - average speed
  - arrival timestamp or elapsed duration context
- If the cockpit provides enough data, the screen may also show cosmetic flight context such as max altitude reached.
- The screen must include a restrained celebratory treatment.
- The screen must provide clear primary actions for Continue Journey and Return Home.
- The screen should be usable on a small phone without hiding the action buttons below the fold.

### 2. Continue Journey
- When the user selects Continue Journey, the app must treat the landed airport as the next temporary origin.
- The user’s saved home airport must remain unchanged.
- The app must reset the next flight setup so the user can choose a new duration and destination from the landed airport.
- The transition should feel like one flight ending and another beginning, not like a reset of the whole app.

### 3. Return Home
- When the user selects Return Home, the app must use the saved home airport as the next origin.
- The home airport should be preselected in the next setup flow.
- If no home airport exists, the app must handle the failure gracefully and return the user to the normal setup path.

### 4. Completed Flight Data
- On arrival, the app must capture a `CompletedFlight` payload.
- The payload must include at least:
  - `booking`
  - `origin`
  - `destination`
  - `originalDestination`
  - `durationSeconds`
  - `distanceKm`
  - `departedAt`
  - `arrivedAt`
  - `wasDiverted`
  - `seat`
- The payload must be serializable and suitable for local persistence or future logbook insertion.
- Completed-flight capture must be idempotent so backgrounding or repeated arrival events do not duplicate the record.

### 5. Navigation and Routing
- The landed experience must be a first-class Expo Router route.
- The route must be registered in the root stack in `app/_layout.tsx`.
- The cockpit should transition into the landed flow with `router.replace(...)` or an equivalent non-backtracking path so the user does not return to an ended cockpit.
- The route should be deep-linkable and typed like the rest of the app’s Expo Router screens.

## UX Requirements
- Use a clear hierarchy:
  - celebration or arrival state first
  - airport identity second
  - route summary and stats third
  - primary actions last
- Keep copy short and readable on mobile.
- Support both light and dark themes.
- Keep animation light and performant.
- Use celebratory color and motion sparingly so the screen still feels like a travel app, not a game overlay.
- If a route preview is included, it should be compact and secondary to the actions.

## Data and State Requirements
- Keep the saved home airport separate from the current flight origin.
- Do not overwrite the home airport when the user continues from a landing.
- Make state transitions safe if the app backgrounds or resumes during landing.
- Ensure the completed-flight payload can be reused by the v0.9.0 logbook work without schema churn.
- Preserve enough data to explain diverted flights clearly.

## Success Metrics
- Reduced friction after landing compared with the current alert-based flow.
- Increased rate of second-flight starts in the same session.
- Fewer dead-end exits immediately after arrival.
- No increase in arrival-related navigation bugs or duplicate completion records.

## Dependencies
- `services/flightTimerService.ts` must continue emitting reliable arrival events.
- `context/FlightContext.tsx` must support the post-flight handoff and separate home/current origin concepts.
- `hooks/useHomeAirport.ts` must continue providing the saved home airport for the return-home path.
- `types/flight.ts` already defines `CompletedFlight` and should remain the source of truth for the data contract.
- `app/_layout.tsx` must register the new landed route in the Expo Router stack.

## Risks
- If the landed screen is too visually heavy, it could feel slow immediately after a long flight.
- If continue-journey state mutates the saved home airport, users may lose their home base.
- If completed-flight data is captured only after navigation, the app may miss a completion during backgrounding.
- If the old alert remains the primary completion UI, the experience will feel duplicated and inconsistent.
- If the landed route is not handled with replace-style navigation, the back stack may let users return to a finished cockpit.

## Proposed Delivery Phases

### Phase 1: Landed Screen Foundation
- [ ] Add a routed landed screen.
- [ ] Show arrival airport identity and route summary.
- [ ] Add completion stats and restrained celebration treatment.
- [ ] Add test IDs for automation and screen verification.

### Phase 2: Continue Journey Flow
- [ ] Persist the landed airport as the next temporary origin.
- [ ] Reset the setup flow so the user can choose a new route from the landed location.
- [ ] Preserve the saved home airport separately.
- [ ] Add graceful fallback handling for missing or unknown airport state.

### Phase 3: Return Home Flow
- [ ] Preselect the saved home airport as the next origin.
- [ ] Return the user to the setup flow with home preloaded.
- [ ] Handle missing home airport gracefully.
- [ ] Surface travel context back home when available.

### Phase 4: Completion Data Handoff
- [ ] Build a reusable `CompletedFlight` payload from cockpit arrival.
- [ ] Store or pass the payload in a way the logbook work can consume later.
- [ ] Ensure arrival handling is idempotent and stable across background/foreground transitions.
- [ ] Remove the old alert as the primary completion experience once the new screen is ready.

## Testing Requirements
- Unit test the state transitions for:
  - cockpit completion
  - continue journey
  - return home
  - missing home airport fallback
- Verify the landed screen renders with realistic completed-flight data.
- Verify navigation from cockpit to landed screen and onward to setup flows.
- Verify theme support in light and dark mode.
- Manually validate the flow on a physical device because this is a mobile-first navigation feature.

## Release Notes Copy
TravelBlock now lands flights with a proper post-flight screen. After a flight completes, users can review the journey, continue from the airport they reached, or return home with one tap.

## Open Questions
1. Should the landed screen show a small route preview map, or should that wait for logbook/map polish?
2. Should Continue Journey reopen the setup flow with the landed airport preloaded, or should it jump directly to destination selection?
3. Should completed-flight persistence land in AsyncStorage first, or should v0.8 already write into the SQLite path intended for v0.9 logbook storage?

## Out of Scope for This Release
- Flight history lists
- Analytics charts
- Export/share features
- Rewards or points economy
- Additional aircraft or transport modes
