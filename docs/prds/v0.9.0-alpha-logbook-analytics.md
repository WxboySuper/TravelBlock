# TravelBlock PRD: v0.9.0-alpha Logbook & Analytics

## Summary
TravelBlock already simulates flights in real time and already has a placeholder Logbook tab, but it does not yet preserve completed flights or help users understand their travel history. This release turns each completed flight into a durable history record, surfaces that history in a mobile-first logbook, and adds lightweight analytics so users can see progress over time.

The goal is to make TravelBlock feel cumulative. A flight should not disappear when the cockpit ends; it should become part of the user’s personal travel record.

## Problem Statement
Today, the app has three separate pieces that do not yet connect:

1. `app/flight/cockpit.tsx` and `services/flightTimerService.ts` complete flights in real time, but completion is not persisted as a reusable history record.
2. `app/(tabs)/explore.tsx` is a Logbook placeholder that only shows an empty state.
3. `services/storageService.ts` persists active setup data, but there is no dedicated history API or queryable logbook store.

As a result, completed flights vanish after landing, and the user has no place to review progress, inspect routes, or understand patterns in their focus sessions.

## Current State
The repository already has several foundations for this feature:

- `types/flight.ts` defines `CompletedFlight`, `FlightBooking`, `Seat`, and the active-flight types.
- `types/storage.ts` already reserves `StorageKey.COMPLETED_FLIGHTS`, which gives us a named storage contract for history.
- `services/storageService.ts` provides a SQLite-backed key-value wrapper through `expo-sqlite/kv-store`.
- `context/FlightContext.tsx` persists active flight setup state, but not completed history.
- `app/(tabs)/explore.tsx` renders an empty Logbook tab using `components/logbook/EmptyLogbook.tsx`.
- `app/_layout.tsx` currently registers the tab navigator and flight flows, but no logbook detail routes or analytics screens.
- `ROADMAP.md` marks v0.9.0 as the Logbook & Analytics release, with logbook, analytics, map, and export work still open.

## Goals
- Persist every completed flight in a durable local history store.
- Show a usable logbook list instead of the current empty placeholder.
- Let users inspect a single flight in detail.
- Surface basic analytics about completed flights, distance, duration, and frequency.
- Support route/map visualization for past flights.
- Keep the logbook fast and readable on small mobile screens.
- Make the history data contract stable enough for future rewards, export, or cross-feature reuse.

## Non-Goals
- Do not build cloud sync or multi-device account storage.
- Do not add social sharing as a core requirement.
- Do not redesign the cockpit or post-flight flow.
- Do not introduce the rewards/points economy.
- Do not replace the existing booking or boarding experience.
- Do not require an always-on network connection.

## User Stories
1. As a user, I want to see every flight I have completed so I can track progress.
2. As a user, I want to open one flight and inspect its route, timing, and outcome.
3. As a user, I want to filter or sort my history so I can find specific sessions.
4. As a user, I want to see simple summaries of how much I have flown and how often.
5. As a user, I want my history to remain on device even if I close the app.

## Functional Requirements

### 1. Completed Flight Persistence
- When a flight completes, the app must create a completed-flight record from the cockpit arrival data.
- The record must be saved locally on device before the user leaves the completion flow.
- The history write must be idempotent so repeated arrival events, app backgrounding, or route re-entry do not duplicate entries.
- The stored record must preserve the source booking, final destination, original destination if diverted, seat, timestamps, duration, and distance.
- The logbook data model must be serializable and stable enough for future export or analytics.

### 2. Logbook Screen
- The Logbook tab must display a list of completed flights once history exists.
- Each list item must show at minimum:
  - origin airport
  - destination airport
  - date or relative time
  - duration
  - distance
- The screen must preserve the current empty state for first-run users.
- The screen must support scroll performance suitable for a growing history list.
- The screen must remain readable in both light and dark themes.
- The screen must provide a clear way to open a flight detail view.

### 3. Flight Detail View
- The user must be able to open a completed flight and inspect the full record.
- The detail view must show:
  - origin and final destination
  - original destination when diverted
  - departure and arrival timestamps
  - duration and distance
  - seat assignment
  - booking metadata that helps identify the flight
- If route visualization is available, the detail view should include a compact route map or route summary.
- The detail view must be dismissible without losing the list position.

### 4. Analytics
- The app must present a lightweight analytics summary for the completed-flight history.
- At minimum, analytics should include:
  - total flights completed
  - total distance flown
  - total focus time translated into flight time
  - average flight duration
  - most recent flight information
- If enough history exists, the analytics view may also show trends such as flights per week or top destinations.
- Analytics should be fast to compute on device and should not block the logbook list.

### 5. Route and Map Visualization
- The history experience should support a route map or route preview for completed flights.
- The route display should reuse the app’s existing geography and map capabilities instead of inventing a separate mapping system.
- Route visualization must be secondary to the core list and detail experience; it should not make the logbook feel heavy.

### 6. Export Readiness
- The logbook data model must be structured so future export/share work can serialize it without schema churn.
- Export UI is not required to be first-class in the initial slice, but the underlying data should be export-safe.

### 7. Navigation and Routing
- The Logbook experience must stay inside Expo Router conventions.
- The existing `app/(tabs)/explore.tsx` route should remain the tab entry point for the logbook overview.
- Detail and analytics views should be reachable through real routes or clearly scoped modal navigation, not through ad hoc global state.
- Navigation from cockpit completion into history should not create broken back-stack behavior.

## UX Requirements
- Keep the empty state encouraging, not dismissive.
- Use concise copy and strong hierarchy for mobile readability.
- Prefer summary cards, compact rows, and short labels over dense paragraphs.
- Make history browsing feel calm and archival, not gamified.
- Keep analytics visually simple and legible on smaller screens.
- Ensure touch targets are large enough for one-handed use.
- Support both themes with no loss of contrast or semantic emphasis.

## Data and State Requirements
- Use a dedicated local history store for completed flights, separate from active-flight setup state.
- Preserve `CompletedFlight` as the source of truth for completed-session data.
- Add a stable unique identifier for history records if needed for deduplication and detail routing.
- Make persistence append-only unless the user explicitly deletes a record later.
- Compute analytics from stored history, not from ephemeral cockpit state.
- Keep home airport and active flight state separate from history records.
- Avoid mutating historical entries when the user starts a new flight.

## Proposed Architecture
- Primary history store: SQLite-backed logbook repository, using the installed `expo-sqlite` dependency.
- Active setup store: keep using the existing storage and context patterns for live flight state.
- History access layer: a dedicated `services/logbookService.ts` module that encapsulates create/query/update/delete behavior.
- UI layer:
  - logbook overview tab
  - flight detail view
  - analytics screen or analytics section
  - optional route/map preview components
- Data contract layer: `types/logbook.ts` or equivalent additions to represent persisted logbook records and aggregated stats.

## Success Metrics
- Users can see completed flights after landing without manual action.
- Users can retrieve a past flight in under a few taps.
- The empty state is replaced by visible history for returning users.
- Analytics render quickly and feel trustworthy.
- No duplicate logbook records appear from a single completed flight.
- Logbook interactions remain smooth as the number of flights grows.

## Dependencies
- `services/flightTimerService.ts` must continue emitting reliable completion events.
- `app/flight/cockpit.tsx` or the post-flight handoff must hand off a completed-flight payload.
- `types/flight.ts` must remain the canonical completed-flight contract.
- `services/storageService.ts` and the SQLite wrappers provide the local persistence foundation.
- `app/(tabs)/explore.tsx` is the current entry point for the logbook tab.
- `app/_layout.tsx` will need to register any new detail or analytics routes.

## Risks
- If persistence is bolted on only after navigation, the app may miss a completion when the user backgrounds or closes the app.
- If the logbook schema is too minimal, analytics and export will require churn later.
- If history calculation happens on every render, the logbook could feel slow on lower-end devices.
- If route visualization is treated as core instead of optional, the first release may become too visually heavy.
- If completed-flight writes are not idempotent, arrival events could create duplicate records.

### Phase 1: History Foundation
- [ ] Create the logbook repository/service.
- [ ] Persist completed flights from the cockpit handoff.
- [ ] Define the persisted logbook record shape and unique identifier strategy.
- [ ] Add tests for create, dedupe, and read paths.

### Phase 2: Logbook List
- [ ] Replace the empty tab with a real completed-flight list.
- [ ] Preserve the empty state for first-run users.
- [ ] Add basic sorting and/or simple filtering if it fits the slice.
- [ ] Add tests for rendering and state transitions.

### Phase 3: Flight Details
- [ ] Add a flight detail view or modal.
- [ ] Show route summary, timestamps, seat, and diversion context.
- [ ] Connect list items to the detail route.
- [ ] Add tests for navigation and detail rendering.

### Phase 4: Analytics and Route Views
- [ ] Add aggregate stats for total flights, distance, and focus time.
- [ ] Add trend summaries when history depth allows.
- [ ] Add optional route/map preview visualization.
- [ ] Add tests for analytics calculations and route rendering.

## Testing Requirements
- Unit test completed-flight persistence and deduplication.
- Unit test analytics aggregation from sample histories.
- Test that the logbook renders correctly when empty and when populated.
- Test navigation from the logbook list to flight details.
- Test that diverted and non-diverted flights render distinct detail states.
- Verify theme support in light and dark mode.
- Manually validate the flow on a physical device because this is a mobile-first history experience.

## Release Notes Copy
TravelBlock now remembers your flights. The new Logbook shows completed sessions, lets you review individual flights, and surfaces simple analytics about your focus travel over time.
